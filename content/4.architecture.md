The concept of the builder based inversion of control architecture for parsers has been introduced in [our previous work](cite:cites modular-parsing).
In this work, we go into more detail on the architecture implemented in Traqula, and how we extended the
architectural vision towards query generation and transformation.
We explain how Traqula exists of many small, interlinkable packages,
and provide more detail on the core package which contains the language agnostic builders and transformers.


### Builder-based Inversion of Control

At the core of Traqula's flexibility is its builder-based inversion of control architecture.
Each functional component, such as a parser rule, generator rule, or transformation step, is declared under a name,
and refers to other components by name rather than by direct implementation.
This indirection allows users to replace or patch implementations simply by changing the mapping from names to functions.
[](#rulemap) (left) illustrates this idea,
while [](#rulemap) (right) shows Traqula's TypeScript definition of the `iri` parser and generator components that rely on named subcomponents.

To manage and share these name-implementation mappings, Traqula uses the [builder design pattern](https://refactoring.guru/design-patterns/builder).
Builders construct complex, modular objects step by step, and expose a uniform API for composing, extending, and patching components.
Traqula provides three task-specific builders: a parser builder, a generator builder, and a general indirection builder.
All share the same underlying name-to-implementation structure, differing only in what the final `build` method produces.

Each builder exposes the following operations (here "rules" and "components" are interchangeable):

- **create**: create a builder either as a copy from another builder, or from a list of initial rules.
- **widen context**: widen the type of the context available to all rules in the builder; context being an alternative to `this`.
- **type patch**: adjust or override the TypeScript types of selected rules, useful when a builder patch changes the exposed API.
- **patch rule**: patches a single already registered rule with a new implementation.
- **add rule**: add a new rule under a new name.
- **add many**: add several new rules at once.
- **delete rule**: remove a rule from the builder.
- **get rule**: retrieve teh current component associated with a name, reflecting any type patches or context changes.
- **merge**: merge another builder into the current one, comparable to [ANTLR's grammar imports](https://github.com/antlr/antlr4/blob/857fb46e781ce9c40249b9f0156c67051cec12c1/doc/grammars.md#grammar-imports).
- **build**: create the final parser, generator, or generic indirection object.

The type-level operations are essential for the maintainability of a modular ecosystem.
When one package patches or extends another builder, type patches allow the exposed API to remain consistent and safe even after invasive modifications.

Each task-specific builder also provides specialised helper functions appropriate for its domain,
visible as the capitalised calls in [](#rulemap).
Parser builders offer combinators such as `CONSUME`, `OR`, `OPTION`, and `MANY` for expressing Chevrotain-style LL(k) grammars.
Generator builders provide formatting helpers such as `PRINT`, `NEW_LINE`, and `HANDLE_LOC` to support structured and round-tripping code generation.
The generic builder exposes only the core indirection helper `SUBRULE`, used to invoke named components.


### Modular Packages

To maximize independent evolution, Traqula is composed of many small, interlinked packages rather than a single big monolithic package.
As a user, you only need to depend on what you actually use.
For example, if you only require SPARQL 1.1 parsing, you can depend solely on _'@traqula/parser-sparql-1-1'_,
without needing to consider the SPARQL 1.2 parser or the generator packages.
This modular approach is typical when flexibility and extensibility are important.
Modularity also encourages Traqula's package APIs to remain open for extension, since Traqula itself relies on those same package interfaces.
For maintainability and version control, all packages are managed in a single GitHub repository under the Comunica organisation:
[https://github.com/comunica/traqula](https://github.com/comunica/traqula).


### Language-Agnostic Core Package

Following the modular package design, Traqula provides a central language-agnostic core package containing the generic builders and transformers.
These components are designed to be reusable across different query languages,
whether for dialects of SPARQL or entirely different languages such as SHACL-compact or GQL.
The core package underpins Traqula's flexibility, enabling parsers, generators, and transformations to be composed in a language-independent manner.

A key feature of the core package is the generic transformer/visitor,
which facilitates AST- and algebra-level transformations/ visiting.
This transformer is type-safe, leveraging TypeScript's generics to adapt dynamically to the AST or algebra representation being manipulated.
Beyond enabling rewrites and translations between query languages,
the transformer also supports AST-level reformatting, since reformatting can be expressed as a transformation on the AST level in the case of Traqula.

To use the transformer, one creates a new transformer object, optionally providing a default transformation context and specifying the types of AST nodes.
Transformations can be guided through preVisitors, which return a transformation context and allow fine-grained control over the process by specifying whether to:
