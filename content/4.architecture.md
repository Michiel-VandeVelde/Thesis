To manage and share these name-implementation mappings, Traqula uses the [builder design pattern](https://refactoring.guru/design-patterns/builder)<span class="comment" data-author="RT">Can you cite something instead of only providing a link? Note also that links by default don't show up in print mode, unless you use the `.mandatory` css class</span>.
Builders construct complex, modular objects step by step, and expose a uniform API for composing, extending, and patching components.
Traqula provides three task-specific builders: a parser builder, a generator builder, and a general indirection builder.
All share the same underlying name-to-implementation structure, differing only in what the final `build` method produces.

- **create**: create a builder either as a copy from another builder, or from a list of initial rules.
- **widen context**: widen the type of the context available to all rules in the builder; context being an alternative to `this`.
- **type patch**: adjust or override the TypeScript types of selected rules, useful when a builder patch changes the exposed API.
- **patch rule**: patches a single already registered rule with a new implementation.
- **add rule**: add a new rule under a new name.
- **add many**: add several new rules at once.
- **delete rule**: remove a rule from the builder.
- **get rule**: retrieve the current component associated with a name, reflecting any type patches or context changes.
- **merge**: merge another builder into the current one, comparable to [ANTLR's grammar imports](https://github.com/antlr/antlr4/blob/857fb46e781ce9c40249b9f0156c67051cec12c1/doc/grammars.md#grammar-imports).

Each task-specific builder also provides specialised helper functions appropriate for its domain,
visible as the capitalised calls in [](#rulemap).
Parser builders offer combinators such as `CONSUME`, `OR`, `OPTION`, and `MANY` for expressing Chevrotain-style LL(k) grammars.
Generator builders provide formatting helpers such as `PRINT`, `NEW_LINE`, and `HANDLE_LOC` to support structured and round-tripping code generation.
The generic builder exposes only the core indirection helper `SUBRULE`, used to invoke named components.


### Modular Packages

To maximize independent evolution<span class="comment" data-author="RT">Not sure independent evolution is the right argument here. I would suggest buiding the story around package and bundle size (where the latter matters a lot in web apps!).</span>, Traqula is composed of many small, interlinked packages rather than a single big monolithic package.
As a user, you only need to depend on what you actually use.
For example, if you only require SPARQL 1.1 parsing, you can depend solely on _'@traqula/parser-sparql-1-1'_,
without needing to consider the SPARQL 1.2 parser or the generator packages.
This modular approach is common in the JavaScript ecosystem when flexibility and extensibility are important.
Modularity also encourages Traqula's package APIs to remain open for extension, since Traqula itself relies on those same package interfaces.
For maintainability and version control, all packages are managed as a [monorepo](https://monorepo.tools/){:.mandatory} in a single GitHub repository under the Comunica organisation:
[https://github.com/comunica/traqula](https://github.com/comunica/traqula).


### Language-Agnostic Core Package

Following the modular package design, Traqula provides a central language-agnostic core package containing the generic builders and transformers.
These components are designed to be reusable across different query languages,
whether for dialects of SPARQL or entirely different languages such as SHACL-compact or GQL.
The core package underpins Traqula's flexibility, enabling parsers, generators, and transformations to be composed in a language-independent manner.
